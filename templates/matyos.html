Getting Started

Introduction

MatyOS is a proof-assistant programming language where formal verification is first-class syntax, not an afterthought. Theorems, axioms, and QED live at the same level as variables and functions. The language is built on a Lexer → Parser → Semantic Analyzer → Interpreter pipeline written entirely in Python.

The source file extension is .el (for Elnamic Language). Programs are case-insensitive for keywords — VAR, var, and Var are all valid. Identifiers support alphanumeric characters and underscores.

Comments use {{ }} for multi-line blocks and // for single-line, keeping the aesthetic close to traditional proof notation.

Design Philosophy Proof is syntax, not ceremony. A theorem that cannot be proven is a compile-time concern, not a runtime surprise.

Installation

Clone the repository and use Python 3.8+ to run programs. No external dependencies are required for the core language runtime.

1

Clone the repository

git clone https://github.com/MatyOS-Project/MatyOS.git
cd MatyOS
2

Install (Mac / Linux — one-liner)

chmod +x install.sh && ./install.sh
# adds `el` to ~/.local/bin — restart shell after
3

Or install via pip (editable)

pip install -e .
# pip install -r requirements.txt  (optional extras)
4

Run a program

el run hello_world.el
# or: python -m compiler.main

Language Reference

Variables & Constants

Variables are declared with var, constants with const. Constants are immutable after declaration — the semantic analyzer enforces this at compile time.

ALGORITHM Vars {
  // Variable declaration (mutable)
  var counter   : integer = 0;
  var pi        : real    = 3.14159;
  var greeting  : string  = "hello";
  var flag      : boolean = true;
  var uncertain : boolean = realistic;

  // Constant declaration (immutable)
  const MAX_SIZE : integer = 100;

  // Assignment and output
  counter = counter + 1;
  show counter;
}

Type System

MatyOS is statically typed. The type of a variable is declared at definition time and enforced by the semantic analyzer. The type system includes the unique realistic three-valued boolean.

The realistic value extends classical boolean logic with a third state representing uncertainty or incomplete knowledge. Useful for modeling real-world conditions where truth is not yet determined.

ALGORITHM BoolDemo {
  var confirmed  : boolean = true;
  var confidence : boolean = realistic;

  if confirmed {
    show "confirmed";
  } elif confidence {
    show "uncertain";
  } else {
    show "rejected";
  }
}

Functions

Functions are declared with the function keyword, optionally accepting typed parameters and returning typed values. Recursive functions are supported.

ALGORITHM Functions {
  // Function declaration — params grouped as (a, b: type)
  function add(a, b : integer) {
    return a + b;
  }

  // Recursive function
  function factorial(n : integer) {
    if n <= 1 {
      return 1;
    }
    return n * factorial(n - 1);
  }

  show add(3, 4);         // 7
  show factorial(5);    // 120
}

Proof System

Formal Verification

The proof system is MatyOS's defining feature. Formal mathematical proofs are written directly in source code. The semantic analyzer links theorems to their proofs and verifies structural correctness. The interpreter executes the proof steps and prints a verified trace.

The system draws from the axiomatic method: you state self-evident truths (axioms), declare propositions (theorems), and provide structured proofs using hypotheses, tests, and QED.

⊢
axiom
A self-evident truth that requires no proof. Foundation of all reasoning in the system.
∀
theorem
A proposition declared to be provable. Must be matched by a corresponding proof block.
{ }
proof
Block that provides the structured argument for a theorem. Contains hypotheses and tests.
∃
hypothesis
An assumed truth scoped within the proof block. Forms the basis of the argument.
?
test
An inline verification step. Tests that a hypothesis holds against declared axioms.
∎
QED
Seals and terminates the proof. Quod Erat Demonstrandum — "which was to be demonstrated."

Axioms

Axioms are self-evident truths. They require no proof. Every axiomatic system begins with a set of axioms that serve as the foundation for all theorems.

ALGORITHM Axioms {
  // Identity axiom
  axiom identity : true === true;

  // Law of excluded middle
  axiom excluded_middle : true or !true;

  // Named with definition for documentation
  definition truth_is_stable : true === true;

  show "axioms registered";
}

Theorems

A theorem declares a proposition. The semantic analyzer requires each theorem to be matched by a proof block of the same name. A theorem without a proof is a compile-time error.

ALGORITHM Theorems {
  theorem truth_is_consistent : true;
  theorem conjunction_holds   : true and true;
  show "theorems declared";
}

Proof Blocks

A proof block contains the structured argument for a theorem. It consists of hypotheses (assumed truths) and tests (verification steps), terminated with QED.

ALGORITHM ProofDemo {
  axiom identity        : true === true;
  axiom excluded_middle : true or !true;

  theorem simple_theorem : true;

  proof simple_theorem {
    hypothesis h1 : true;          // assume h1 holds
    test verify_h1 : h1 : true;    // verify against axioms
    QED;                           // ∎ proof complete
  }

  show "Theorem proved.";
}

Realistic Logic

The realistic keyword introduces a third boolean state — uncertain. Classical logic operates on a closed world assumption; realistic logic acknowledges incomplete information.

ALGORITHM Realistic {
  var sensor_reading : boolean = realistic;

  // Three-way branching with realistic state
  if sensor_reading === true {
    show "confirmed";
  } elif sensor_reading === false {
    show "denied";
  } else {
    show "uncertain";
  }
}
Bring Format

The Bring Data Format

Bring is a human-readable data serialization format designed for configuration files and package manifests. It is similar in spirit to TOML but with a simpler syntax and native schema support. MatyOS uses .bring files for package configuration and dependency management.

Bring is parsed by the bring-parser library and accessible from El programs via the bring import system.

Syntax

Bring files use key = value pairs. Values can be primitives, objects, or arrays. Objects use {} blocks and arrays use []. Attributes annotate values with @attr = val.

Primitives

name = "MatyOS"
version = 1
pi = 3.14
active = true

Objects

author {
  name = "Daftyon"
  email = "dev@daftyon.io"
}

Arrays

tags = ["math", "proof", "logic"]
ports = [8080, 8443, 9000]

Attributes

@required = true
@type = "string"
name = "MatyOS"

Complete Example

package.bring 
name    = "my-project"
version = "1.0.0"
stable  = true

author {
  name  = "Dev"
  email = "dev@example.com"
}

dependencies = ["math_utils", "neural_networks"]

config {
  debug   = false
  timeout = 30
  tags    = ["production", "stable"]
}

Schemas

Bring supports inline schema declarations using the schema keyword. Schemas define the structure of objects and their types, enabling validation at parse time.

schema Package {
  name    : string
  version : string
  stable  : bool
}

schema Author {
  name  : string
  email : string
}

name    = "my-project"

Packages & CLI

Package System

MatyOS uses the bring keyword for importing packages. Packages are resolved from the MatyOS package registry (Easier Hub) and cached locally. The from and as keywords enable named and aliased imports.

ALGORITHM PackageDemo {
  // Import full package
  bring math_utils;

  // Import with alias
  bring neural_networks as nn;

  // Named imports from package
  bring { Vector, Matrix } from math_package;

  // Scoped import from registry
  bring linear_algebra from scientific_hub;

  show "packages loaded";
}

Package Manifest

Package dependencies are declared in a .bring file at the project root. The CLI installs and caches all declared dependencies.

deps.bring 
name = "my_project"

[dependencies]
math_utils       = "^1.2.0"
neural_networks  = "^0.9.1"

Architecture

Compiler Pipeline

Every El program passes through four stages. Each stage can raise a typed error (LexerError, ParserError, SemanticError) with line and column information.

Lexer → Parser → Semantic Analyzer → Interpreter

Lexer

Converts raw source text into a token stream. Case-insensitive keyword matching. Supports {{ }} multi-line comments and // single-line. Recognizes ===, !=, >=, <= as compound operators. Handles single- and double-quoted strings.

Parser

~1150-line recursive descent parser. Builds a full AST for all language constructs: variables, constants, functions, objects, control flow, proofs (axiom / theorem / proof / hypothesis / test / QED), and package imports.

Semantic Analyzer

Multi-pass AST visitor. Resolves scope chains and symbol tables. Enforces constant immutability. Links each theorem to its corresponding proof block. Detects undeclared identifiers and type mismatches before execution.

Interpreter

Tree-walking interpreter. Evaluates all AST nodes including proof steps, package imports (Easier Hub), built-in show, involve, and object access. Proof execution prints a colored verification trace to stdout: [AXIOM], [THEOREM], [HYPOTHESIS], [TEST ✓], [QED ∎].
